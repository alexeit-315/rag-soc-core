Вот пошаговый процесс работы с Git в CLI для разработчика и тимлида:

***

## 1. Подключение разработчика к существующему репозиторию

```bash
# Клонирование удалённого репозитория на локальную машину
git clone <URL_репозитория>

# Переход в директорию проекта
cd <имя_репозитория>

# (Опционально) Настройка имени и email для коммитов
git config user.name "Имя Фамилия"
git config user.email "email@example.com"
```

***

## 2. Создание разработчиком ветки для работы над задачей

```bash
# Обновление локальной главной ветки (main/master)
git checkout main
git pull origin main

# Создание и переключение на новую ветку для задачи
git checkout -b feature/<название-задачи>
# или современным синтаксисом:
# git switch -c feature/<название-задачи>
```

***

## 3. Обновление разработчиком новой ветки в удалённом Git

```bash
# Внесение изменений в файлы проекта...

# Добавление изменённых файлов в индекс
git add .
# или конкретные файлы: git add файл1 файл2

# Создание коммита с описанием
git commit -m "feat: описание выполненной работы"

# Отправка ветки в удалённый репозиторий
git push -u origin feature/<название-задачи>
# флаг -u устанавливает отслеживание удалённой ветки
```

***

## 4. Передача разработчиком результатов на проверку (Code Review)

```bash
# Создание Pull Request (PR) / Merge Request (MR)
# Эта команда работает только при наличии установленного GitHub CLI:
gh pr create --title "Описание задачи" --body "Детали реализации"

# Альтернативно — через веб-интерфейс GitHub/GitLab/Bitbucket:
# 1. Открыть репозиторий в браузере
# 2. Нажать "New Pull Request" / "Create Merge Request"
# 3. Выбрать base: main, compare: feature/<название-задачи>
# 4. Отправить на ревью тимлиду
```

***

## 5. Слияние веток тимлидом после проверки

```bash
# Тимлид обновляет локальную главную ветку
git checkout main
git pull origin main

# (Опционально) Локальное слияние для проверки
git merge --no-ff feature/<название-задачи>

# Если конфликтов нет — слияние через веб-интерфейс:
# 1. Открыть Pull Request в браузере
# 2. Проверить статус CI/CD и комментарии ревьюеров
# 3. Нажать "Merge pull request" → "Confirm merge"

# Альтернативно через CLI (GitHub CLI):
gh pr merge <номер-PR> --merge --delete-branch
# --merge — обычное слияние, --delete-branch — удалить ветку после мержа

# Удаление локальной ветки после слияния
git branch -d feature/<название-задачи>

# Обновление главной ветки
git pull origin main
```

***

### Краткая шпаргалка по основным командам

| Действие | Команда |
|----------|---------|
| Клонировать репозиторий | `git clone <URL>` |
| Создать и переключить ветку | `git checkout -b <ветка>` |
| Добавить файлы | `git add .` |
| Сделать коммит | `git commit -m "сообщение"` |
| Отправить ветку | `git push -u origin <ветка>` |
| Создать PR (GitHub CLI) | `gh pr create` |
| Слить PR (GitHub CLI) | `gh pr merge <номер>` |
| Удалить локальную ветку | `git branch -d <ветка>` |

Этот процесс обеспечивает изоляцию задач, код-ревью и чистую историю коммитов в проекте. [sky](https://sky.pro/wiki/profession/git-osnovnye-komandy-i-principy-raboty/)
