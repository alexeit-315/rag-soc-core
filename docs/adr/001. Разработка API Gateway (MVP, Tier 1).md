## Контекст

### Предпосылки

В рамках разработки RAG SOC System требуется создать единую точку входа в Test-контур для обеспечения аутентификации, авторизации, маршрутизации и контроля доступа к микросервисам. На этапе MVP (Tier 1) API Gateway должен обеспечить взаимодействие с тремя ключевыми сервисами: Converter, Indexer, Experimentation Service.

### Архитектурные ограничения

- API Gateway является **единственной точкой входа** в Test-контур для всех внешних клиентов и внутренних сервисов (кроме прямых вызовов Experimentation Service → RAG Orchestrator)
- API Gateway должен быть доступен как внутри кластера K8s, так и извне
- Admin UI и Web UI являются клиентами API Gateway (браузерные запросы от инженеров и пользователей)
- В архитектуре присутствует **Istio Service Mesh**, который может обеспечивать circuit breaker, retry, observability на уровне сервисов
- Converter уже имеет OpenAPI спецификацию (converter-api.yaml) с полным набором эндпоинтов
- **Язык реализации API Gateway: Kotlin** (согласно I-01)

## Решение

### Выбор технологического стека для MVP

Выбран **custom API Gateway на Kotlin** с использованием Spring Cloud Gateway или Ktor.

| Критерий | Spring Cloud Gateway (Kotlin) | Ktor | Kong (отклонен) |
|----------|-------------------------------|------|-----------------|
| Язык | Kotlin (I-01) | Kotlin (I-01) | Не Kotlin/Java/Python |
| Контроль над кодом | Полный | Полный | Ограниченный |
| Интеграция с Keycloak | Spring Security OAuth2 | Через плагины | Готовая |
| Rate limiting | Встроенный | Требуется реализация | Встроенный |
| Логирование аудита | Полный контроль | Полный контроль | Ограниченный |
| Кастомизация | Высокая | Высокая | Низкая |
| Ресурсоемкость | Средняя (JVM) | Низкая | Средняя |
| Соответствие I-01 | Да | Да | Нет |

**Плюсы custom решения на Kotlin:**
- Полное соответствие требованиям I-01 (Kotlin)
- Полный контроль над функциональностью
- Возможность реализации специфических требований (trace mode, многоуровневое логирование)
- Интеграция с существующим стеком Spring (если используется)
- Отсутствие дополнительных компонентов (Kong)

**Минусы custom решения для MVP:**
- Требуется написание и поддержка кода
- Rate limiting требует реализации (можно использовать Bucket4j)
- Health checks для upstream требуют реализации

**Альтернативы, отклоненные на MVP:**
- *Kong API Gateway* — сложность внесения изменений
- *Istio Ingress Gateway* — не предоставляет встроенных механизмов аутентификации и rate limiting, невозможна трассировка внутри сервисов

### Взаимодействие с сервисами

На этапе MVP API Gateway взаимодействует с сервисами только по синхронному протоколу (REST). Буферизация и распараллеливание операций производится на уровне сервисов.

**Статус операции:**
- API Gateway не подписывается на Kafka события
- Клиент самостоятельно опрашивает `/status` эндпоинт целевого сервиса
- API Gateway проксирует запросы статуса

### Взаимодействие с Kafka

API Gateway отправляет логи в Kafka для централизованного сбора и анализа.

| Топик | Формат сообщения | Назначение |
|-------|------------------|------------|
| `logs` | Structured JSON (см. пример в разделе Логирование) | Сбор логов всех сервисов → Loki |

### Аутентификация и авторизация

#### Решение для MVP: JWT валидация в API Gateway + ролевая модель

**Механизм:**
1. API Gateway получает JWT токен из заголовка `Authorization: Bearer <token>`
2. Валидация токена (подпись, срок действия, audience) с использованием публичного ключа Keycloak
3. Извлечение claims: `resource_access.rag-system.roles` (роли), `email` (идентификатор), `sub` (user_id)
4. Авторизация на основе ролей для каждого маршрута

**Предварительная ролевая модель для MVP:**

| Роль                | Доступные операции                                                                                                                          | Соответствующий сервис                     |
|---------------------|---------------------------------------------------------------------------------------------------------------------------------------------|--------------------------------------------|
| `converter_manager` | Загрузка комплекта исходных документов, управление конвертацией комплектов (запуск и остановка конвертации, получение статуса конвертации)  | Converter                                  |
| `indexer_manager`   | Загрузка комплекта обработанных документов, управление индексацией комплектов (запуск и остановка индексации, получение статуса индексации) | Indexer                                    |
| `ml_engineer`       | Загрузка и управление экспериментами                                                                                                        | Experimentation Service                    |
| `user`              | Взаимодействие с инференсами системы (`POST /api/v1/chat`, `POST /api/v1/soc/*`, `GET /api/v1/search`) (будущий этап)                       | RAG Orchestrator, Batch Processor (future) |
| `admin`             | Все операции (full access)                                                                                                                  | Все сервисы                                |

**Расширение ролевой модели (post-MVP):**
- **Авторизация на уровне комплектов документации (SEC-02):** добавление claim `allowed_document_sets` с маппингом `document_set_id → role`
- **Разделение ролей:** `document_viewer` (только чтение), `experiment_viewer` (только просмотр результатов), `system_monitor` (доступ к метрикам)
- **Динамическая авторизация:** запрос к сервису авторизации для проверки сложных политик
- **API Key для внешних систем:** отдельный механизм аутентификации с собственным скоупом

**Плюсы MVP-решения:**
- Stateless — нет дополнительных запросов к Keycloak
- Полный контроль над логикой авторизации
- Легко расширяется

**Минусы MVP-решения:**
- Отзыв токена требует уменьшения TTL (компромисс между безопасностью и UX)
- Роли зашиты в JWT — изменение роли требует выдачи нового токена

**Целевой этап (post-MVP):**
- Использование `introspection` endpoint Keycloak для мгновенного отзыва токенов
- Интеграция с сервисом авторизации для проверки доступа на уровне комплектов документации

#### Обработка запросов без токена

| Тип запроса | Поведение | Обоснование |
|-------------|-----------|-------------|
| Браузерный запрос к `/admin*` или `/` | Редирект на Keycloak login (302) | Пользователь ожидает интерактивного входа |
| API запрос без токена (любой другой путь) | `401 Unauthorized` | Безопаснее — не раскрываем существование эндпоинтов неавторизованным клиентам |

### Различение внешних и внутренних запросов

**Решение:** Использование комбинации маршрутизации и network policies.

| Тип запроса | Идентификация | Применяемые политики                                                                           |
|-------------|---------------|------------------------------------------------------------------------------------------------|
| Внешний (браузер, CI/CD) | Запрос приходит с IP вне K8s cluster | Rate limiting (5 req/min для экспериментов, 10 req/min для конвертации), полная аутентификация |
| Внутренний (между сервисами) | Запрос приходит из K8s CIDR | **Ослабленный rate limiting или отсутствует**, аутентификация через mTLS (Istio)               |

**Важное уточнение:** Внутренний вызов между сервисами Experimentation Service → RAG Orchestrator является исключением и **не проходит через API Gateway** (решение требует подтверждения, единственное исключение не ласт значительного выигрыша в производительности, но усложняет реализацию). Это обеспечивается:
- Настройкой K8s Network Policies
- Использованием внутренних Service имен (например, `orchestrator.rag-test.svc.cluster.local`)


### Обработка отказа целевых сервисов

**Проблема:** Converter, Indexer или Experimentation Service могут зависнуть, работать медленно или быть недоступны.

**Решение для MVP: Комбинация Istio + API Gateway Health Checks**

```
┌─────────────────────────────────────────────────────────────────┐
│                   API Gateway (Kotlin)                          │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │  Health Check Manager                                       ││
│  │  - Active health checks (периодические запросы к /health)   ││
│  │  - Passive health checks (на основе ошибок)                 ││
│  │  - Upstream pool management (исключение нездоровых)         ││
│  └─────────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼ (HTTP запросы через Istio)
┌─────────────────────────────────────────────────────────────────┐
│                         Istio                                   │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────────────┐  │
│  │   Circuit   │───▶│   Retry     │───▶│   Timeout           │  │
│  │   Breaker   │    │ (exp backoff)│   │                     │  │
│  └─────────────┘    └─────────────┘    └─────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │ Target Service  │
                    │ (Converter/     │
                    │  Indexer/Exp)   │
                    └─────────────────┘
```

**Компоненты решения:**

| Уровень | Механизм | Конфигурация | Ответственность |
|---------|----------|--------------|-----------------|
| API Gateway | Active Health Check | Проверка `/health` каждые 5 сек, порог 3 неудачных попытки | Исключение нездоровых инстансов из пула upstream |
| API Gateway | Passive Health Check | На основе 5xx ошибок (5 ошибок в 60 сек) | Автоматическое исключение проблемных инстансов |
| API Gateway | Retry (application level) | `attempts: 2`, только для GET запросов `/status` | Повторные попытки при временных сбоях |
| API Gateway | Timeout | `connectTimeout: 5s`, `readTimeout: 60s` | Предотвращение зависания запросов |
| API Gateway | Fallback Response | При недоступности всех upstream → `503 Service Unavailable` | Информирование клиента |
| Istio | Circuit Breaker | `maxConnections: 5, maxPendingRequests: 10, maxRequests: 3` | Защита от перегрузки целевого сервиса |
| Istio | Retry (infrastructure) | `attempts: 2, perTryTimeout: 30s, retryOn: "5xx,reset,connect-failure"` | Повторные попытки на уровне сетевых сбоев |

**Обработка длительных операций (Converter/Indexer):**

API Gateway **не агрегирует статусы** и не реализует бизнес-логику опроса. Это ответственность клиента:
1. Клиент получает `job_id` от Converter/Indexer
2. Клиент самостоятельно опрашивает `/status` эндпоинт
3. API Gateway проксирует запросы статуса без изменений

**Почему Gateway не должен агрегировать:**
- Gateway должен оставаться тонким (stateless, без бизнес-логики)
- Разные клиенты могут иметь разные стратегии опроса
- Статусы операций могут быть сложными (прогресс, промежуточные состояния)

**Что происходит при поломке сервиса:**
1. API Gateway health checks обнаруживают недоступность (active)
2. Или Istio circuit breaker размыкается при ошибках (passive)
3. API Gateway возвращает клиенту `503 Service Unavailable`
4. Gateway логирует ошибку с мета-информацией
5. **Нет автоматического fallback** — это ответственность клиента или Orchestrator'а (не Gateway)

### Rate Limiting для MVP

**Решение:** Rate limiting на уровне API Gateway с использованием Bucket4j.

| Тип запроса | Лимит | Store | Примечание                                       |
|-------------|-------|-------|--------------------------------------------------|
| `/api/v1/convert*` | 10 requests/min per IP | In-memory (Caffeine) | Конвертация — очень ресурсоемкая операция        |
| `/api/v1/index*` | 5 requests/min per IP | In-memory (Caffeine) | Индексация —  ресурсоемкая операция              |
| `/api/v1/experiment*` | 5 requests/min per IP | In-memory (Caffeine) | Эксперименты — для защиты от случайного запуска  |
| `/admin*`, `/` | 60 requests/min per IP | In-memory (Caffeine) | UI — более либеральный лимит                     |

### Canary deployments (A/B тестирование)

Для поддержки требования **S-30** (возможность обновления моделей без остановки системы) API Gateway интегрируется с Istio для организации canary-развертываний и A/B тестирования.

**Механизм для MVP:**

| Компонент | Роль | Конфигурация |
|-----------|------|--------------|
| Istio VirtualService | Маршрутизация трафика между версиями API Gateway | Поэтапное переключение: 10% → 50% → 100% |
| Istio DestinationRule | Определение подмножеств (subsets) версий | `version: v1`, `version: v2` |
| HTTP Header `x-canary-version` | Идентификация экспериментальной группы для A/B тестов | Проброс заголовка от клиента к сервисам |

**Сценарии использования:**

| Сценарий | Описание | Конфигурация трафика |
|----------|----------|----------------------|
| **Canary deployment** | Постепенное обновление API Gateway | 90% → v1, 10% → v2 |
| **A/B тестирование** | Сравнение двух версий пайплайна | По заголовку `x-canary-version: v2` → 100% на v2 |
| **Rollback** | Откат к предыдущей версии | 100% → v1 |

**Ограничения MVP:**
- Управление canary-развертываниями осуществляется через ручное изменение Istio CRD (не автоматизировано)
- Метрики сравнения версий собираются через Prometheus, но дашборд для A/B тестирования — post-MVP
- Поддержка только для API Gateway; для RAG Orchestrator и Generator — следующий этап

**Пример конфигурации (описательно):**
- VirtualService определяет два маршрута: default (90% трафика на стабильную версию) и canary (10% трафика на новую версию)
- При успешном мониторинге (error rate < 1%, latency p95 < 45 сек) доля canary увеличивается до 100%
- При обнаружении проблем трафик автоматически переключается на стабильную версию через Istio

**Post-MVP улучшения:**
- Автоматизация canary на основе метрик (Argo Rollouts или Flagger)
- Интеграция с дашбордом для визуализации A/B тестов
- Распространение canary на RAG Orchestrator и Generator
 
### Мониторинг и трассировка

**Решение для MVP: интеграция с Prometheus (метрики) + Jaeger (трассировка)**

| Компонент | Назначение | Реализация                                                     |
|-----------|------------|----------------------------------------------------------------|
| **Prometheus** | Сбор метрик | Micrometer + `/actuator/prometheus` endpoint                   |
| **Jaeger** | Распределенная трассировка | Spring Cloud Sleuth + Brave + `traceparent` header propagation |
| **Loki** | Сбор логов | Логи в Kafka → Promtail → Loki                                 |

**Метрики для сбора (S-10, S-11):**

| Метрика | Тип | Описание |
|---------|-----|----------|
| `gateway_requests_total` | Counter | Общее количество запросов (method, path, status) |
| `gateway_requests_duration_seconds` | Histogram | Длительность запросов (p50, p95, p99) |
| `gateway_errors_total` | Counter | Количество ошибок по типу (auth, timeout, 5xx) |
| `gateway_upstream_health` | Gauge | Статус здоровья upstream сервисов (1=healthy, 0=unhealthy) |
| `gateway_rate_limit_hits` | Counter | Количество срабатываний rate limiting |

**Трассировка (S-23):**

| Режим | Поведение | Доступность |
|-------|-----------|-------------|
| **Normal** | Включена базовая трассировка (Jaeger), логирование мета-информации | Всегда |
| **Trace mode** | Включена полная трассировка (Jaeger), логирование тел запросов/ответов | Только для авторизованных пользователей с ролью `ml_engineer` + флаг `?trace=true` |

### Логирование

**Решение:** Многоуровневое логирование с отправкой в Loki через stdout.

| Уровень | Логируемая информация | Формат | Назначение |
|---------|----------------------|--------|------------|
| **ERROR** | Ошибки аутентификации, 5xx, circuit breaker events | Structured JSON | Оповещения, отладка проблем |
| **WARN** | Превышение rate limiting, недоступность upstream (но health check еще не исключил), таймауты, повторные попытки | Structured JSON | Предупреждения, требующие внимания |
| **INFO** | Метод, path, user_id (из JWT), status, duration_ms, client_ip, user_agent | Structured JSON | Аудит (S-27), анализ использования |
| **DEBUG** | + Тело запроса (только при `?trace=true`), тело ответа (при ошибках) | Structured JSON | Отладка экспериментов |

**Включение DEBUG:**
- Через query parameter: `?debug=true` или `?trace=true`
- Только для авторизованных пользователей с ролью `ml_engineer`

### Развертывание в Kubernetes

API Gateway разворачивается в Kubernetes в неймспейсе `rag-test` с использованием Helm-чарта.

**Конфигурация развертывания:**

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Реплики | 2 (min), до 5 (max) | Отказоустойчивость (R-07) + горизонтальное масштабирование |
| Liveness probe | `/actuator/health/liveness`, порог 3, период 10 сек | Проверка живости процесса (S-05) |
| Readiness probe | `/actuator/health/readiness`, порог 2, период 5 сек | Проверка готовности принимать трафик (S-05) |
| Resources (requests) | CPU: 250m, Memory: 512Mi | Гарантированные ресурсы для стабильной работы |
| Resources (limits) | CPU: 500m, Memory: 1Gi | Ограничение потребления на ПК разработчика (P-30) |
| Service type | ClusterIP (внутренний) + Ingress (внешний) | Доступность внутри и снаружи K8s |

**Health check endpoints:**

| Endpoint | Назначение | Возвращает |
|----------|------------|------------|
| `/actuator/health/liveness` | Проверка, жив ли процесс (K8s liveness probe) | 200 OK если процесс работает |
| `/actuator/health/readiness` | Проверка, готов ли сервис принимать трафик (K8s readiness probe) | 200 OK если все upstream сервисы доступны, иначе 503 |

**Управление конфигурацией:**
- Конфигурация через ConfigMap (маршруты, лимиты)
- Публичный ключ Keycloak (для валидации JWT) хранится в ConfigMap, так как не является секретной информацией. 
- Параметры окружения через Helm values

**Для MVP использование Kubernetes Secrets не требуется. При переходе к introspection-валидации токенов (post-MVP) потребуется добавление Secrets для client secret.**

**Исполняемый файл:**
- Сборка в Docker образ на основе JVM (Kotlin/Spring Boot)
- Образ хранится в Harbor Container Registry
- Версионирование через Git tags + CI/CD pipeline

## Альтернативы

### Альтернатива 1: Kong API Gateway

| Критерий | Оценка                                                                                      |
|----------|---------------------------------------------------------------------------------------------|
| Соответствие I-01 | Нет (язык не Kotlin/Java/Python)                                                            |
| Контроль над кодом | Низкий                                                                                      |
| Кастомизация trace mode | Сложная                                                                                     |
| **Вердикт** | Отклонено — не соответствует подходу с гибкой настройкой и полным контролем над API Gateway |

### Альтернатива 2: Istio Ingress Gateway + Envoy Filter для аутентификации

| Критерий | Оценка                                       |
|----------|----------------------------------------------|
| Сложность | Высокая (требуется Lua или WASM для OIDC)    |
| Rate limiting | Через Envoy, но сложнее в настройке          |
| Аутентификация | Нет встроенной OIDC, требуется custom filter |
| **Вердикт** | Отклонено — избыточная сложность             |

### Альтернатива 3: Только Istio + без Gateway (прямые вызовы)

| Критерий | Оценка |
|----------|--------|
| Аутентификация | Нет единой точки — каждый сервис проверяет JWT |
| Rate limiting | Распределенный, сложный |
| Логирование аудита | Распределенное, сложное |
| **Вердикт** | Отклонено — не соответствует требованию единой точки входа |

## Последствия

### Позитивные последствия

1. **Унификация безопасности** — единая точка для аутентификации и авторизации
2. **Снижение сложности сервисов** — Converter, Indexer, Experimentation Service не занимаются auth
3. **Гибкость маршрутизации** — можно менять бэкенды без изменения клиентов
4. **Масштабируемость** — API Gateway может быть горизонтально масштабирован
5. **Observability** — единое логирование, метрики и трассировка всех запросов
6. **Rate limiting** — защита ресурсов Test-контура от случайных перегрузок
7. **Полный контроль** — код на Kotlin, легкая кастомизация
8. **Соответствие I-01** — использование разрешенного языка Kotlin

### Негативные последствия

1. **Разработка и поддержка кода** — требует ресурсов команды
2. **Latency overhead** — ~5-10 мс на запрос (приемлемо для требований P-01, P-02)
3. **In-memory rate limiting** — при горизонтальном масштабировании лимиты независимы (для MVP приемлемо)
4. **Не все эндпоинты реализованы в MVP** — чат, SOC, поиск отложены
5. **Health checks реализуются вручную** — дополнительный код

### Риски и митигации

| Риск | Вероятность                                                                             | Митигация                                                                         |
|------|-----------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------|
| API Gateway становится узким местом | Низкая (при 1 RPS)                                                                      | Горизонтальное масштабирование при росте нагрузки                                 |
| Ошибки в кастомной авторизации | Средняя                                                                                 | Юнит-тесты, интеграционные тесты                                                  |
| In-memory rate limiting не работает при scale-out | Низкая (для MVP)                                                                        | Для post-MVP — переход на Redis                                                   |
| JWT токен украден | Средняя | Короткий TTL (15 мин), mTLS между сервисами, возможность отзыва через blacklist   |
