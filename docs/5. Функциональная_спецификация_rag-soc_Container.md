| `converter_manager` | Загрузка комплекта исходных документов, управление конвертацией (запуск, остановка, статус) | Converter |

#### Обработка запросов без токена

| Тип запроса | Поведение | HTTP статус |
|-------------|-----------|-------------|
| Браузерный запрос к `/admin*` или `/` | Редирект на Keycloak login | 302 |
| API запрос без токена (любой другой путь) | Отказ в доступе | 401 Unauthorized |

#### Upstream Health Checks

API Gateway реализует active и passive health checks для сервисов.

| Тип | Механизм | Конфигурация |
|-----|----------|--------------|
| Active | Периодический запрос к `/health` | Каждые 5 сек, порог 3 неудачных попытки |
| Passive | На основе 5xx ошибок | 5 ошибок в 60 сек |

При недоступности всех upstream инстансов API Gateway возвращает `503 Service Unavailable`.

#### Исключение: прямые вызовы Experimentation Service (под вопросом)

Experimentation Service вызывает RAG Orchestrator **напрямую**, минуя API Gateway.

| Вызывающий | Целевой сервис | Причина | Аутентификация |
|------------|----------------|---------|----------------|
| Experimentation Service | RAG Orchestrator | Снижение latency, изоляция экспериментов от внешних политик (rate limiting) | mTLS (Istio) |

## 6. Data Flow Notes

### 6.1 API Gateway Flow (Test)

1. Запрос поступает в API Gateway
2. Проверка JWT токена через IdP (OIDC)
3. Применение rate limiting (Redis)
4. Маршрутизация к соответствующему сервису:
    - `/api/v1/chat` → RAG Orchestrator
    - `/api/v1/soc/*` → Batch Processor
    - `/api/v1/experiment` → Experimentation Service
    - `/api/v1/dataset` → Dataset Service 
    - `/api/v1/dictionary` → Dictionary Service 
    - `/api/v1/version/compare` → Version Comparison Service
    - `/api/v1/document/upload` → Converter
   - `/api/v1/index/*` → Indexer
    - `/admin/*` → Admin UI
5. Логирование запроса (Loki)
6. Сбор метрик (Prometheus)

### 6.2 API Gateway Flow (Prod)

1. Запрос поступает в API Gateway
2. Проверка JWT токена через IdP (OIDC)
3. Применение rate limiting (Redis)
4. Маршрутизация к соответствующему сервису:
    - `/api/v1/chat` → RAG Orchestrator
    - `/api/v1/soc/*` → Batch Processor
    - `/api/v1/search` → Search Service
    - `/admin/*` → Admin UI
5. Логирование запроса (Loki)
6. Сбор метрик (Prometheus)

### 6.3 External API Flow (Prod)

1. Внешняя система отправляет запрос с API Key
2. External API Gateway проверяет rate limiting
3. API Key Manager валидирует ключ в PostgreSQL
4. При успешной валидации запрос перенаправляется в Internal API Gateway
5. Ответ возвращается внешней системе
6. Логирование с идентификацией клиента (API Key ID)

### 6.4 Индексация (только в Test)

1. Converter загружает документы из doc_sources
2. Converter сохраняет JSON в Object Storage
3. Converter публикует событие в Kafka
4. Indexer получает событие из Kafka
5. Indexer вызывает Embedding Service
6. Embedding Service сохраняет векторы в Vector DB
7. Embedding Service индексирует текст в Elasticsearch
8. Готовые индексы реплицируются в Production:
    - Snapshot Vector DB → Object Storage → Prod Vector DB
    - Snapshot Elasticsearch → Object Storage → Prod Elasticsearch

### 6.5 Обработка запроса в Test (с Istio)

1. API Gateway → RAG Orchestrator (через Istio)
2. Orchestrator → Query Processor (через Istio с circuit breaker)
3. Query Processor → Dictionary Service (через Istio)
4. Orchestrator → Index Router (через Istio)
5. Index Router → Search Service (через Istio)
6. Search → Vector DB и Elasticsearch (через Istio с retry)
7. Search → Reranker (через Istio)
8. Reranker → Generator (через Istio)
9. Generator → Inference Gateway (через Istio)
10. Inference Gateway → LLM Provider (через Istio с circuit breaker и retry)
11. Результат кэшируется и возвращается

### 6.6 Обработка запроса в Production (с Istio)

1. API Gateway → RAG Orchestrator (через Istio)
2. Orchestrator → Query Processor (через Istio)
3. Orchestrator → Index Router (через Istio)
4. Index Router → Search Service (через Istio)
5. Search → Vector DB и Elasticsearch (через Istio с retry)
6. Search → Reranker (через Istio)
7. Reranker → Generator (через Istio)
8. Generator → Inference Gateway (через Istio)
9. Inference Gateway → LLM Provider (через Istio с circuit breaker и retry)
10. Результат кэшируется и возвращается

### 6.7 SOC Batch Processing (Test и Prod)

1. Batch Processor получает список вопросов
2. Для каждого вопроса проверяет Retrieval Cache (или Cache)
3. При попадании — использует кэшированный контекст
4. При промахе — вызывает Orchestrator
5. Сохраняет результаты в Generation Cache (или Cache)
6. Возвращает агрегированный ответ

### 6.8 Health Checks

- Каждый сервис предоставляет:
    - `/health/live` — liveness probe (проверка, жив ли процесс)
    - `/health/ready` — readiness probe (проверка, готов ли сервис принимать трафик)
- K8s использует эти эндпоинты для оркестрации

### 6.10 Обработка запросов без аутентификации

1. Запрос поступает в API Gateway
2. Проверка наличия JWT токена:
   - Если путь начинается с `/admin` или `/` и токен отсутствует → редирект на Keycloak (302)
   - Если путь начинается с `/api/` и токен отсутствует → `401 Unauthorized`
3. При наличии токена выполняется его валидация (подпись, срок, audience)
4. При невалидном токене → `401 Unauthorized`

### 6.11 Rate Limiting

1. API Gateway извлекает client_ip из запроса (X-Forwarded-For или remote address)
2. Проверка лимита для path:
   - `/api/v1/convert*` → 10 запросов в минуту
   - `/api/v1/index*` → 5 запросов в минуту
   - `/api/v1/experiment*` → 5 запросов в минуту
   - `/admin*`, `/` → 60 запросов в минуту
3. При превышении лимита → `429 Too Many Requests`
4. При ошибке Redis (если используется) → логирование WARN, пропуск запроса (fail-open)

### 6.12 Trace Mode

1. API Gateway проверяет наличие параметра `?trace=true` и роль пользователя `ml_engineer`
2. При выполнении условий:
   - Устанавливается заголовок `traceparent` для downstream сервисов
   - Включается DEBUG уровень логирования для данного запроса
   - Логируются тела запросов и ответов
3. Трассировка отправляется в Jaeger
4. При невыполнении условий — нормальный режим работы
