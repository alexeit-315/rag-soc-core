# Структура архитектурных документов 
## Этап 1: Определение требований и границ системы
1. Требования к системе в формате FURPS+
- Functionality (Функциональность):
  - Конвертация документов (PDF, DOCX → JSON/TXT/MD)
  - Индексация и поиск по эмбеддингам
  - Генерация ответов на опросные листы
  - Управление промптами и моделями
- Usability (Удобство использования):
  - Web UI для конечных пользователей
  - API для интеграции
  - Административный интерфейс для ML-инженеров
- Reliability (Надежность):
  - Отказоустойчивость (K8s, репликация)
  - Резервное копирование данных
  - Восстанавливаемость после сбоев
- Performance (Производительность):
  - Время ответа RAG < 3 сек (p95)
  - Скорость индексации > 100 страниц/мин
  - Поддержка > 100 одновременных пользователей
- Supportability (Поддерживаемость):
  - Логирование и мониторинг
  - Документация API
  - On-call процессы
- + (Дополнительные ограничения):
  - Design constraints: K8s, микросервисная архитектура
  - Implementation constraints: Python для AI, Go для высоконагруженных сервисов
  - Interface constraints: REST/gRPC API
  - Physical constraints: On-premise или облако, требования к GPU
2. Архитектурное видение системы (Architectural Vision)
- Высокоуровневое описание целей архитектуры
- Ключевые архитектурные характеристики (масштабируемость, безопасность, тестируемость)
- Основные архитектурные подходы (микросервисы, событийно-ориентированная, data mesh)
- Принципы: API-first, Infrastructure as Code, Security by Design
-  Ограничения и допущения
- Критерии успеха архитектуры
## Этап 2: Фиксация архитектурных решений
3. ADR (Architecture Decision Records)  
Набор коротких markdown-файлов, фиксирующих каждое ключевое архитектурное решение (выбор БД, выбор формата обмена, стратегия кеширования, модель безопасности и т.д.). Каждый ADR содержит контекст, рассмотренные альтернативы и обоснование выбора.
- ADR по стратегии работы с GPU и очередями
  - Контекст: Один GPU (GTX-1650 4GB) на продакшн. Нужно обслуживать и синхронные запросы чата (ожидание < 30 сек), и асинхронные batch (SOC до 500 вопросов).
  - Рассмотренные альтернативы: Параллельная обработка (не влезает в VRAM). Последовательная обработка с FCFS (будет блокировать чат).
  - Решение: Внедрить два независимых рантье (queues) или отдельных инференс-пода: priority для чата (max 1 concurrent) и batch для SOC (max 1 concurrent, низкий приоритет). Использование модели в квантованном формате (Q4_K_M).
  - Обоснование: Обеспечивает предсказуемую latency для интерактивных пользователей ценой увеличения общего времени обработки SOC.
- ADR по стратегии хранения и версионирования индексов
  - Контекст: Требования D-06, D-07, D-09, D-10 (версионирование, откат, метаданные парсера).
  - Решение: Принять соглашение об именовании индексов: idx_<dataset_id>_v<major>_<minor>. В Object Storage хранить дампы индексов и index_manifest.json (содержит hash датасета, версию парсера, версию модели эмбеддингов, дату создания). В PostgreSQL хранить маппинг "активный индекс для dataset_id".
  - Альтернативы: Хранить всё в векторной БД без версионирования (не соответствует D-06).
4. Стандарты разработки и технические соглашения (Development Standards)  
Артефакт, фиксирующий соглашения и стандарты разработки, которые обеспечивают единообразие реализации. Лежит в основе Code Review, Onboarding и настройки линтеров/форматтеров в CI/CD . Он обязателен к исполнению для всех членов команды.
5. Функциональная_спецификация  
Описывает требуемые характеристики системы (функциональность). Определяет функциональные требования, включая рабочие процессы, поведение системы, входные и выходные данные, а также взаимодействие с точки зрения пользователя.
6. Контекстная диаграмма (C4 Model - Level 1)  
Описывает систему как "черный ящик", её пользователей (операторы, внешние API, инженеры) и внешние зависимости (системы аутентификации, внешние репозитории моделей).
7. Контейнерная диаграмма (C4 Model - Level 2)  
- Показывает высокоуровневую структуру программного обеспечения: какие сервисы (контейнеры) мы создаем (API Gateway, RAG Orchestrator, Indexer, Converter), какие системы хранения используем (БД, Object Storage) и как они взаимодействуют.
- Результат: Схема взаимодействия доменов и компонентов. 
8. Use Cases / User Flows
- Chat flow
- SOC upload flow
- Converting/Indexing flow
- Admin flow
- Experiment flow
9. Диаграмма последовательностей Sequence Diagrams (C4 Model - Level 3-lite)
- Chat request (RAG pipeline)
- SOC batch processing
- Indexing pipeline
10. Диаграмма развертывания (Deployment Diagram)  
Описывает инфраструктуру. Два контура (Staging/Prod), компоненты K8s (Ingress, Services, Deployments, ConfigMaps, Secrets), политики сетевой безопасности (Network Policies), взаимодействие с внешними хранилищами (S3-compatible storage) и механизмы обновления (Helm charts).
11. Спецификация API (OpenAPI/Swagger)  
Описание контрактов взаимодействия между компонентами (синхронные REST/gRPC вызовы) и событий (асинхронная шина, если потребуется).
12. Service Communication & Networking Specification
- Общая модель взаимодействия
- Service-to-Service Communication
- Service Discovery (Istio)
- Traffic Management (Istio)
- Network Policies (K8s)
- mTLS (Istio)
- External Access
- Timeouts / Retries (ОЧЕНЬ важно)
- Rate Limiting
- Kafka Communication
14. Модель данных (Data Model)  
- Схема JSON для "машинообрабатываемого формата" (чревато расширением).
- Схема метаданных (ключевые поля: версия документа, дата индексации, источник, хэш, права доступа).
- Схема БД для истории запросов и Feature Store.
14. Документ по безопасности (Security/Threat Model)  
Описание стратегии авторизации (RBAC/ABAC), шифрования данных (in-transit и at-rest), управления секретами (Vault/K8s Secrets), процессов безопасной разработки (SCA/SAST/DAST), аудита действий.
15. Документ по тестированию (Test Strategy)  
Описание типов тестов: юнит-тесты конвертеров, интеграционные тесты индексатора, end-to-end тесты RAG на тестовых данных, benchmark тесты производительности.
16. CI/CD Pipeline Definition (GitLab CI / Jenkins Pipeline as Code)  
Описание стадий пайплайна: сборка, модульное тестирование, сканирование кода (SAST), сборка образа, сканирование образа (SCA), развертывание в тестовый контур, интеграционное тестирование, нагрузочное тестирование, развертывание в прод (blue/green или canary).
17. Параметризация и конфигурация (Helm Charts)  
Набор Helm-чартов с четко описанными значениями (values.yaml) для каждого окружения, позволяющий параметризовать все: от версий моделей до количества реплик и порогов срабатывания RAG.
18. MVP Scope Definition & Phased Rollout Plan
- Определение требований к MVP (Tier 1) /  Tier 2 / Tier 3 (subset FURPS)
- Какие функции явно НЕ входят (out of scope)
- Пользовательские сценарии MVP (User Stories)
- Критерии приемки MVP (Exit Criteria):    
- Roadmap (MVP (Tier 1) -> Tier 2 -> Tier 3)
