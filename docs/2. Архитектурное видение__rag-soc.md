## 1. Основные положения

Система предназначена для ускорения работы инженеров, архитекторов, аналитиков и пресейл-команд при поиске информации и автоматизации заполнения SOC (Statement of Compliance).

Поддерживает:
- on-prem инфраструктуру
- возможность масштабирования GPU
- мульти-языковые корпуса (RU/EN/возможен CN)
- контроль качества моделей и метрик
- два изолированных контура: Test (R&D) и Production
- Service Mesh (Istio) для service discovery, circuit breaker, retry, mTLS
- **External API для интеграции с внешними системами**

## 2. Мотивация (Описание проблемы)

### 2.1 Боль пользователей

- Поиск информации в больших объемах документации занимает от нескольких минут до **2–4 часов** на вопрос
- Контекстный поиск неэффективен:
    - разная терминология у вендоров
    - большое количество нерелевантных документов
    - необходимость ручной фильтрации

### 2.2 Последствия

- Потеря времени инженеров и аналитиков
- Снижение качества ответов
- Потеря доверия заказчика
- Риск потери тендеров

### 2.3 Пользователи

- пресейл-инженеры (основные)
- архитекторы
- support
- аналитики
- **внешние системы (через API)**

### 2.4 Критичные сценарии

- **Чат** — ~90% обращений
- **SOC (опросники)** — ~10% обращений, но критичны по бизнес-ценности и объемны по количеству вопросов
- **API-интеграция** — для подключения к корпоративным системам (CRM, порталы)

## 3. Цели проекта

### 3.1 SOC (Statement of Compliance)

- Сокращение времени обработки: **с 3–4 дней → до 1 дня**
- Ручная проверка: **2–4 часа**
- Точность: **>90%**

### 3.2 Сложные вопросы (архитекторы)

- Время ответа: **≤ 1 минуты**
- Точность: **>90%**

### 3.3 Дополнительные цели

- Поддержка multilingual (RU/EN/возможен CN)
- Автоматизация тестового контура:
    - эксперименты
    - метрики
    - сравнение моделей
- Возможность сравнения версий комплектов документации
- **Предоставление REST API для интеграции с внешними системами**

## 4. Архитектурный контекст

### 4.1 Контуры системы

#### Test (R&D)

- Эксперименты: модели, промпты, retrieval pipeline
- Автоматическое тестирование: quality, e2e, безопасность (DAST)
- Управление словарями и онтологией
- Сравнение версий документации
- Оборудование: ПК разработчика (16GB RAM, GTX-1650)

#### Production

- Обработка реальных пользовательских запросов
- Контроль качества и latency
- Стабильная конфигурация pipeline
- Выбор индексов на основе метаданных
- Rollback индексов
- **Предоставление External API**
- Оборудование: сервер Core i5-2400, 32GB RAM, GTX-1650 (требуется апгрейд)

### 4.2 Инфраструктура

- Полностью **on-prem**
- Интернет-доступ: GitHub, Vulnerability Databases
- Данные: NFS / S3 (объектное хранилище)
- Оркестрация: Kubernetes
- Service Mesh: Istio (service discovery, circuit breaker, retry, mTLS, observability)
- CI/CD: GitLab CI + Helm

### 4.3 Интеграции

- Внутренние системы:
    - IdP (Keycloak) — OIDC/OAuth2
    - Application Security Suite (SAST/SCA/DAST)
    - мониторинг / логирование (Prometheus/Grafana/Loki/Jaeger/Kiali)
- **Внешние системы (через External API):**
    - CRM
    - Корпоративные порталы
    - Системы документооборота
- API: REST

### 4.4 LLM инфраструктура

- Локальный кластер (Ubuntu)
- Модель: Llama (on-prem)
- Поддержка квантованных моделей (GGUF, GPTQ, AWQ)
- Inference Gateway для управления очередью GPU и rate limiting
- Circuit Breaker на вызовах LLM (через Istio)

### 4.5 Аутентификация

- OIDC / OAuth2
- Отдельный IdP (Keycloak)
- **API Key для внешних интеграций** (опционально)

### 4.6 Мониторинг и логирование

- Prometheus + Grafana (метрики)
- Loki (логи)
- Jaeger (трейсинг)
- Kiali (service mesh observability)
- Дополнительно: audit логирование, хранение истории запросов и ответов
- Сбор метрики cost-per-query (GPU-секунды)
- **Мониторинг использования External API (RPS, latency, ошибки)**

## 5. Описание базовой архитектуры

### 5.1 UI Layer

- **Web UI** — чат, поиск, отображение источников
- **Admin UI** — управление экспериментами, конфигами, словарями
- **API Gateway** - единая точка входа, аутентификация, авторизация, rate limiting, маршрутизация, логирование, метрики, трассировка
- **External API** — REST API для внешних систем (CRM, порталы, автоматизация)

### 5.2 Control Plane (Test)

- Prompt Store (версионирование промптов)
- Config Store (конфигурации пайплайнов)
- Dictionary Store (словари и онтология)
- Query Processor (нормализация, расширение, режимы strict/balanced)
- Dictionary Service (разрешение терминов)
- Experimentation Service (оркестрация экспериментов)
- Evaluation Service (оценка и выбор модели)
- Dataset Service (управление датасетами)
- Version Comparison Service (сравнение версий документации)

### 5.3 Data Plane (Test)

- Converter (конвертация документов)
- Indexer (индексация)
- Embedding Service (генерация эмбеддингов)
- Index Router (выбор индекса, версионирование, rollback)
- Search Service (гибридный поиск: semantic + BM25)
- Retrieval Cache / Generation Cache (Redis)
- Batch Processor (SOC обработка с reuse retrieval)
- Vector DB (Chroma) — отдельные БД на комплект
- Elasticsearch — BM25 поиск
- Object Storage (S3/NFS)
- Kafka (события пайплайна, асинхронные запросы)
- PostgreSQL (метаданные, конфиги, история, метрики)

### 5.4 Compute Plane (Test)

- RAG Orchestrator (управление пайплайном, trace, fallback)
- Reranker (переранжирование)
- Generator (генерация ответа с цитированием)
- Inference Gateway (очередь GPU, rate limiting)

### 5.5 Control Plane (Production)

- Feature Store (единый — промпты, конфиги)
- Query Processor (нормализация + расширение)
- Dictionary Service (онтология)

### 5.6 Data Plane (Production)

- Index Router (выбор индекса по метаданным, rollback)
- Search Service (гибридный поиск: semantic + BM25)
- Cache (Redis) — единый кэш для retrieval и generation
- Batch Processor (SOC обработка с reuse retrieval)
- Vector DB (Chroma) — отдельные БД на комплект
- Elasticsearch — BM25 поиск
- PostgreSQL (конфиги, история, метрики)
- Kafka (асинхронные запросы)

### 5.7 Compute Plane (Production)

- RAG Orchestrator (управление пайплайном)
- Reranker (переранжирование)
- Generator (генерация ответа с цитированием)
- Inference Gateway (очередь GPU, rate limiting)

### 5.8 External API Layer (Production)

| Компонент | Ответственность | Входы | Выходы | Зависимости |
|-----------|-----------------|-------|--------|-------------|
| External API Gateway | Предоставление REST API для внешних систем | HTTP/REST запросы | Ответы | API Gateway, IdP, Orchestrator |
| API Key Manager | Управление ключами для внешних интеграций | Запросы на создание/отзыв ключей | API Keys | PostgreSQL |
| Rate Limiter | Ограничение частоты запросов от внешних систем | API запросы | Разрешение/блокировка | Redis |

### 5.9 Service Mesh (Istio)

- Service Discovery — автоматическое обнаружение сервисов в K8s
- Circuit Breaker — защита от каскадных отказов (LLM, Vector DB)
- Retry с exponential backoff — автоматические повторные попытки при временных сбоях
- mTLS — шифрование внутреннего трафика
- Observability — сбор метрик, трейсинга, Kiali для визуализации

### 5.10 CI/CD & DevSecOps

- CI: GitLab CI (сборка, тестирование, SAST/SCA)
- CD: Helm + GitLab CI (деплой, rollback)
- Security: SAST/SCA (в CI), DAST (на staging)
- Liveness/Readiness Probes: каждый сервис предоставляет для корректного оркестрирования

### 5.11 Репликация индексов

- Индексы создаются только в Test контуре
- Готовые индексы реплицируются в Production:
    - Через Object Storage (S3 snapshots)
    - Репликация Chroma через файловую систему

## 6. Итоги анализа проблемы

### 6.1 Ограничения

- Малый FTE: **0.05–0.1**
- Ограниченные GPU ресурсы на старте (требуется апгрейд для Production)
- Latency: **≤ 60 сек**

### 6.2 Риски

- Низкое качество ответов → потеря доверия
- Ошибки в SOC → потеря тендеров
- Галлюцинации LLM
- Нерелевантный retrieval
- Каскадные отказы при падении LLM или Vector DB (минимизируются через Circuit Breaker)
- **Злоупотребление External API** (минимизируется через rate limiting и API keys)

### 6.3 Надежность

- Режим работы: **8/5**
- Downtime: допустим
- Backup: раз в неделю
- RTO: **8 часов**
- RPO: **1 час**
- Circuit Breaker на критических внешних вызовах
- Retry с exponential backoff
- **Rate limiting на External API**

## 8. Технические требования к качеству системы

### 8.1 Performance

- p50: ≤ 30 сек
- p95: ≤ 45 сек
- p99: ≤ 60 сек
- **External API p95: ≤ 5 сек**

### 8.2 Scalability

- Добавление GPU
- Горизонтальное масштабирование (K8s)
- Service Mesh обеспечивает балансировку нагрузки
- **External API масштабируется независимо**

### 8.3 Reliability

- SLA: **~99%**
- Бэкапы (еженедельно)
- Возможность отказоустойчивости (в будущем)
- Circuit Breaker на LLM и Vector DB
- Retry с exponential backoff
- **Rate limiting на External API (100 req/min per client)**

### 8.4 Supportability

- Команда: **0.05–0.1 FTE**
- Мониторинг: latency, ошибки, GPU usage, LLM quality, cost-per-query
- Kiali для визуализации сервис-меша
- **Мониторинг External API: RPS, latency, ошибки, usage per API key**

### 8.5 Security & Compliance

- audit логирование
- RBAC (разграничение доступа на уровне комплектов)
- on-prem deployment
- Разграничение доступа к K8s на основе неймспейсов (административное требование)
- mTLS между сервисами через Istio
- **API Key аутентификация для External API**
- **Rate limiting для защиты от DDoS/злоупотреблений**
